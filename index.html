Relational Database Fundamentals

The relational model, introduced by E.F. Codd in 1970, is based on predicate logic (verifying statements as true/false) and set theory (manipulating groups of data).

Integrity Rules

Ensuring data quality

Entity Integrity

All primary key entries must be unique and NOT NULL. Ensures every row has a unique identity.

Referential Integrity

Foreign key values must match an existing primary key or be NULL. Prevents orphan records.

üí°

Flags

Use special codes like 'NS' (No Section) instead of NULL to avoid null-handling issues, with a matching dummy record in the referenced table.

Normalization

Eliminating redundancy

Process of organizing data to reduce redundancy and anomalies (insertion, update, deletion problems).

First Normal Form

  • No repeating groups (multi-valued attributes)
  • Primary key identified
  • Each cell contains atomic (single) values

Second Normal Form

  • Must be in 1NF
  • All non-key attributes fully depend on the entire primary key
  • No partial dependencies (relevant for composite keys)

Third Normal Form

  • Must be in 2NF
  • No transitive dependencies
  • Non-key attributes depend only on the primary key

Normalization Example

Unnormalized: ADVISER: Santos A. | SUBJECT: Math | STUDENTS: John L., Mae R., Mark V.

1NF: Split multi-valued STUDENTS into separate rows

2NF: Create separate tables: Advisers, Subjects, Students

3NF: Add linking table StudentAdvisers (ADV_ID, SUBJ_CODE, STU_NUM)

üìà

GROUP BY & HAVING

Aggregate by groups

SQL
-- Orders per status
SELECT STATUS, COUNT(*) AS OrderCount 
FROM TBL_ORDERS 
GROUP BY STATUS;

-- Products per category (only categories with 2+ products)
SELECT CATEGORY_ID, COUNT(*) AS ProductCount 
FROM TBL_PRODUCTS 
GROUP BY CATEGORY_ID 
HAVING COUNT(*) >= 2;

-- Total inventory per product
SELECT PRODUCT_ID, SUM(QUANTITY) AS TotalStock 
FROM TBL_INVENTORIES 
GROUP BY PRODUCT_ID;

JOIN Types

Combine tables

INNER JOIN: Only matching rows from both tables

SQL
SELECT o.ORDER_ID, c.NAME, o.STATUS, o.ORDER_DATE
FROM TBL_ORDERS o
INNER JOIN TBL_CUSTOMERS c ON o.CUSTOMER_ID = c.CUSTOMER_ID;

LEFT JOIN: All rows from left table + matches from right (NULL if no match)

SQL
SELECT p.PRODUCT_NAME, c.CATEGORY_NAME
FROM TBL_PRODUCTS p
LEFT JOIN TBL_PRODUCT_CATEGORIES c ON p.CATEGORY_ID = c.CATEGORY_ID;

RIGHT JOIN: All rows from right table + matches from left (NULL if no match)

SQL
SELECT c.CATEGORY_NAME, p.PRODUCT_NAME
FROM TBL_PRODUCTS p
RIGHT JOIN TBL_PRODUCT_CATEGORIES c ON p.CATEGORY_ID = c.CATEGORY_ID;

FULL JOIN: All rows from both tables (NULL where no match)

SQL
SELECT c.NAME, o.ORDER_ID, o.STATUS
FROM TBL_CUSTOMERS c
FULL OUTER JOIN TBL_ORDERS o ON c.CUSTOMER_ID = o.CUSTOMER_ID;

Complex Query Examples

Real-world scenarios

SQL
-- Order summary with customer and employee names
SELECT 
    o.ORDER_ID, 
    c.NAME AS Customer, 
    CONCAT(e.FIRST_NAME, ' ', e.LAST_NAME) AS SalesRep,
    o.STATUS, 
    o.ORDER_DATE
FROM TBL_ORDERS o
INNER JOIN TBL_CUSTOMERS c ON o.CUSTOMER_ID = c.CUSTOMER_ID
INNER JOIN TBL_EMPLOYEES e ON o.EMPLOYEE_ID = e.EMPLOYEE_ID;

-- Order total with product details
SELECT 
    o.ORDER_ID,
    p.PRODUCT_NAME,
    i.QUANTITY,
    i.PRICE,
    (i.QUANTITY * i.PRICE) AS LineTotal
FROM TBL_ORDERS o
INNER JOIN TBL_ITEMS i ON o.ORDER_ID = i.ORDER_ID
INNER JOIN TBL_PRODUCTS p ON i.PRODUCT_ID = p.PRODUCT_ID
ORDER BY o.ORDER_ID, p.PRODUCT_NAME;

-- Total inventory value per warehouse
SELECT 
    w.WAREHOUSE_NAME,
    SUM(inv.QUANTITY * p.LIST_PRICE) AS TotalValue
FROM TBL_INVENTORIES inv
INNER JOIN TBL_WAREHOUSES w ON inv.WAREHOUSE_ID = w.WAREHOUSE_ID
INNER JOIN TBL_PRODUCTS p ON inv.PRODUCT_ID = p.PRODUCT_ID
GROUP BY w.WAREHOUSE_NAME
ORDER BY TotalValue DESC;
